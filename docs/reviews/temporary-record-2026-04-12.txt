CoursePilot v3 主 Agent + SessionState Runtime 升级计划
Summary
Router 升级为主 Agent，负责两件事：mode 判定和执行计划生成。前端传入的 mode 不再是硬决定，而是 mode hint；最终以 Router 的 resolved_mode 为准。
SessionState 升级为服务端持久化的会话状态中心，承担短期记忆、执行进度、Agent 间通信和状态回写；各 Specialist Agent 只通过自己的上下文管理器读写 SessionState。
v3 继续保留 learn / practice / exam 三模式和现有 API 兼容层，但内部执行改为“Lead Router -> Runtime -> Specialist Agent -> State Patch”。
fallback 保留，但必须是“影子校验 + 可观测回退”，不能静默掩盖新链路失效。
Key Decisions
Git 基线：
先提交当前状态，打 tag baseline/pre-v3-2026-04-11，再创建分支 v3/architecture-upgrade。
Python 基线：
统一使用 C:\Users\10423\miniconda3\envs\study_agent\python.exe；禁止用当前默认 python 3.6。
模式判定：
ChatRequest.mode 对外保留，但语义改为 mode_hint。
Router 输出 resolved_mode、mode_reason、plan_version。
Runtime 只按 resolved_mode 执行。
SessionState：
第一版采用服务端持久化优先。
API 新增可选 session_id；没有则服务端创建。
客户端继续传 history 以兼容旧链路，但状态真源切到服务端 SessionState。
Agent 通信：
不再用零散 quiz_meta / exam_meta / history_summary_state 作为主通道。
统一改为 SessionState + StatePatch。
Fallback：
默认走新链路。
仅在关键节点失败时回退到兼容链路，并强制写 telemetry、audit、trace。
测试和灰度环境提供 strict_new_runtime=1，关闭 fallback，确保新架构真实可跑。
Implementation Changes
1. Lead Router
将当前 RouterAgent 升级为 LeadRouterAgent。
输入：message、mode_hint、课程信息、用户画像、最近会话摘要、长期记忆摘要。
输出：PlanPlusV1，至少包含：
resolved_mode、mode_reason、need_rag、need_memory、style、output_format、retrieval_query、memory_query、capabilities、risk_level、permission_mode、replan_policy。
resolved_mode 允许与 mode_hint 不同；若不同，Runtime 记录一次 mode_override 事件。
2. SessionState + Agent Context
新增 SessionStateV1 作为服务端真源，至少包含：
session_id、course_name、requested_mode_hint、resolved_mode、task_full_text、task_summary、current_stage、current_step_index、history_summary、selected_memory、last_quiz、last_exam、tool_audit_refs、permission_mode、fallback_flags、idempotency_keys。
新增 StatePatchV1：
Agent 不直接改全局状态，只返回 patch。
为每个 Specialist Agent 配置独立上下文管理器：
TutorContextManager、QuizContextManager、GraderContextManager。
这些管理器负责：
从 SessionState 中做选择性注入；
控制 token 预算；
从 Agent 输出中提取 StatePatch 并回写。
SessionState 成为 Agent 间通信主通道：
Quiz 出题写入 last_quiz/last_exam；
Grader 读取并更新评分、弱点、当前阶段；
Tutor 读取任务摘要、上下文和历史学习状态。
3. Runtime / TaskGraph
引入 ExecutionRuntime 和 TaskGraphV1。
TaskGraphV1 只允许白名单步骤：
plan_intent、prefetch_rag、prefetch_memory、build_agent_context、detect_submission、run_tutor、run_quiz、run_exam、run_grade、persist_session_state、persist_records、persist_memory、synthesize_final。
Router 决定模式和参数，但不能自由创造新步骤。
submission 检测仍保持代码侧确定性规则，不交给 LLM 自由判断。
Runtime 负责执行顺序、并发、重试、fallback、事件上报、幂等控制。
4. Service Layer
从 Runner 抽出 6 个服务：
RAGService、MemoryService、ContextAssembler、ToolHub、WorkspaceStore、TelemetryService。
RAGService：
管理 retriever 生命周期、课程级缓存、索引可用性检查。
MemoryService：
负责长期记忆检索、写入、画像更新、qa_summary 归档。
ContextAssembler：
负责全局预算和分层注入，不再把混合上下文伪装成“教材”。
ToolHub：
统一工具权限、phase contract、risk level、dedup、idempotency、审计日志。
WorkspaceStore：
统一 notes/practices/exams/mistakes 的保存。
TelemetryService：
统一 trace、fallback 记录、对比指标。
5. API / Compatibility
保留 /chat、/chat/stream。
ChatRequest 新增可选 session_id，保留 mode 但内部按 mode_hint 解释。
/chat 返回中附带 session_id 和 resolved_mode。
/chat/stream 首轮 hidden meta 中回传 session_id、resolved_mode、current_stage。
旧前端若只依赖 history + tool_calls，仍可工作；新前端优先用 session_id 续聊。
旧 quiz_meta / exam_meta / history_summary_state 保留一个兼容周期，仅用于读取，不再作为主写入格式。
6. Fallback Validation
每个关键节点都记录：
path=new|fallback、reason、step_name、session_id、request_id。
